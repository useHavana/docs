The Havana Public API is a **CRM-agnostic, read-only API plus thin webhooks** for writing Havana engagement data back into your own CRM. Havana works your leads across voice, SMS, WhatsApp, and email; this API is how the results — statuses, classified intents, transcripts, appointments, opt-outs — land back in the system your team already lives in.

Every record and every webhook event carries `external_id`: the record ID *your* CRM assigned to the lead, captured when the lead was imported into Havana. You never need to map Havana IDs to your own — upsert on `external_id` and you are done.

## Pull is the backbone, webhooks accelerate

The primary integration pattern is a **daily delta poll**:

1. Call `GET /api/v1/leads?updated_since=<last poll>` and page through results with `cursor`.
2. Upsert each lead into your CRM keyed on `external_id`.
3. Persist the last `updated_at` you processed as the next poll's `updated_since`.

Webhooks exist to **accelerate time-sensitive moments** — a lead opted out, a call just ended, a lead asked for a human. Events are deliberately thin: an event type, the lead's `external_id` and Havana ID, a per-lead monotonic sequence number, and `occurred_at`. They never carry full resource state. On receipt, fetch the referenced resource from the pull API, which is always the source of truth.

<Tip>
  A missed webhook is never data loss. Whatever an event would have told you, your next `updated_since` poll carries the same state.
</Tip>

## Classification is asynchronous

Havana classifies engagements (calls, messages, emails) asynchronously. Every classifiable record carries a `classification_status` of `pending` or `completed` and a nullable `intent`:

- The `call.completed` webhook fires the moment a call ends, with `classification_status: pending` and `intent: null`.
- A `lead.classified` event follows once the classifier lands — typically within minutes.
- Late classifications bump the lead's `updated_at`, so the lead reappears in your next delta poll and the re-poll catches the final intent.

## What's in v1

The v1 surface is read-only plus webhooks:

| Endpoint | What it returns |
| --- | --- |
| `GET /api/v1/leads` | Delta-sync lead list (`updated_since` + cursor) |
| `GET /api/v1/leads/{external_id}` | Single lead: status, latest intent, engagement counts |
| `GET /api/v1/leads/{external_id}/activities` | Calls, messages, emails, and status transitions |
| `GET /api/v1/leads/{external_id}/thread` | Full interleaved transcript across all channels |
| `GET /api/v1/calls/{id}` | Call detail: transcript, recording URL, intent, transfer outcome |
| `GET /api/v1/leads/{external_id}/appointments` | Appointments booked for the lead |

Webhook events: `lead.status_changed`, `lead.classified`, `call.completed`, `lead.unsubscribed`, `handoff.requested`.

## Conventions

- All endpoints live under the `/api/v1` path prefix at `https://api.usehavana.com`.
- Authenticate with a per-team API key: `Authorization: Bearer hav_...`.
- Every list endpoint uses cursor pagination: pass `cursor` and `limit`, read `next_cursor` from the response (`null` means the end).
- Every non-2xx response uses a single error envelope: `{code, message, detail}`.
- Webhook payloads are signed with HMAC-SHA256, delivered in the `X-Havana-Signature` header.
